# Тесты API `/analyze`

Скрипт `validate.py` отправляет кейсы из `test_responses.json` на ваш сервер и печатает в терминал:

- по каждому объявлению строку вида `ID …: (Ждали: 0|1, Получили: 0|1)` — сравнение эталонного и фактического `shouldSplit` (1 = `true`, 0 = `false`);
- блок **Precision / Recall / F1-Score** с подписями «Точность / Полнота / Баланс» по метке `shouldSplit`;
- блок по **дополнительным микрокатегориям** (без исходного `mcId`): TP/FP/FN, micro Precision / Recall / F1 и Accuracy по `shouldSplit`.

## Запуск

1. Поднимите приложение (из корня репозитория):

   ```bash
   uvicorn main:app --reload
   ```

2. В другом терминале:

   ```bash
   python tests/validate.py
   ```

По умолчанию запросы идут на `http://127.0.0.1:8000/analyze`. Другой адрес:

```bash
python tests/validate.py --url http://127.0.0.1:8000/analyze
```

Другой файл с кейсами:

```bash
python tests/validate.py --cases tests/my_cases.json
```

## Формат `test_responses.json`

Массив объектов. У каждого объекта:

| Поле | Смысл |
|------|--------|
| `input` | Тело запроса в Swagger (`itemId`, `mcId`, `mcTitle`, `description`) — то, что вы вставляете в форму. |
| `expected` | Эталонный ответ модели: `detectedMcIds`, `shouldSplit`, `drafts` (как в ответе API). |

Пример:

```json
{
  "input": {
    "itemId": 5001,
    "mcId": 201,
    "mcTitle": "Ремонт квартир и домов под ключ",
    "description": "Делаем ремонт под ключ, отдельно сантехника и электрика."
  },
  "expected": {
    "detectedMcIds": [101, 102, 103],
    "shouldSplit": true,
    "drafts": []
  }
}
```

**Важно:** в метриках по микрокатегориям **исходный** `mcId` из `input` не участвует — он отфильтровывается и в эталоне, и в ответе API.

### Как править эталон

1. Откройте [http://127.0.0.1:8000/docs](http://127.0.0.1:8000/docs), выполните `POST /analyze` с телом из поля `input`.
2. Скопируйте JSON ответа в поле `expected` (подправьте `drafts` при необходимости).
3. Сохраните файл и снова запустите `python tests/validate.py`.

Так вы постепенно накапливаете «золотой» набор ожидаемых ответов под вашу модель и промпт.

### Комментарии в JSON

Стандартный JSON комментариев не поддерживает. Скрипт **игнорирует** строки, целиком состоящие из `// …`, и хвост строки после `//` (вне кавычек), так что можно держать рядом черновик с подсказками. Надёжнее — хранить примеры в этом README или в отдельном `.md` и в репозитории коммитить уже **валидный** JSON.

## Идеи для проверок

- **Регрессия после смены промпта:** прогон по всему файлу и сравнение метрик с прошлым прогоном (вручную или скриптом).
- **Точечные кейсы:** добавьте объявление, где модель ошибалась; зафиксируйте эталон после ручной правки.
- **Граничные случаи:** пустой `detectedMcIds`, `shouldSplit: false` при узком описании; наоборот — несколько услуг в тексте при общей микрокатегории.
- **Юнит-тесты формул метрик** (без сети и GigaChat):

  ```bash
  python tests/validate.py --unit
  ```

## Что означают числа в первом блоке

`Ждали` и `Получили` — это только **флаг разбиения** (`shouldSplit`): `1` соответствует `true`, `0` — `false`. Детальный разбор по ID микрокатегорий смотрите в блоке «Дополнительные микрокатегории».
